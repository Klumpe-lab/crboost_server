# Subtomo Gallery — Session Handoff

**Status:** sprites visually broken. Atlas + index are correct on disk; the
problem is almost certainly **browser caching of the atlas PNG**, not the
CSS sprite math.

This is a handoff doc for the next session. Read this first. It captures

## 1. What the gallery is supposed to do

Per-tomogram subtomo gallery: a CSS sprite-atlas grid where each tile is a
2D thumbnail of one pick (mean of visible tilt frames from
`<i>_stack2d.mrcs`). Click a tile to mark it selected. Sort by score / Z.
Sits alongside a tomogram-preview image in the per-tomo dashboard.

Full pipeline doc: `services/visualization/SUBTOMO_GALLERY.md`.

## 3. What the user sees (the symptom)

Tiles render with **wrong content per tile**. Across multiple attempts at
fixing the CSS, the symptoms shifted:

- **Attempt 1 (initial v9 deploy, 192-px tiles, percentage CSS):** "tiles/sprites
  all seem to not fit into the boxes" — quartered/spliced views.
- **Attempt 2 (percentage CSS with `minmax(144px, 1fr)`):** "I see crescents,
  upper-left corner of the sprite fitting inside the tile" — clearly a
  scale mismatch between bg-image and tile div.
- **Attempt 3 (fixed 192-px tiles, pixel-exact CSS):** "4 circles (cutouts)
  per tile" — strongly suggests the bg image is rendering at HALF the
  expected scale within the tile div, so a 2×2 chunk of source tiles is
  visible per display tile.
- **Attempt 4 (fixed 96-px tiles, pixel-exact CSS, current state):** "STill
  broken." User stopped describing — they were done.

The shifting symptoms across CSS attempts that were each independently
correct is the tell. Something is *not* the CSS.

## 4. Root-cause hypothesis (high confidence)

**The browser is serving a stale, cached atlas PNG.**

Evidence: `main.py:89` returns `/api/vis-asset` files with header
`Cache-Control: public, max-age=86400`. That's 24-hour aggressive caching.

The asset URL is `/api/vis-asset?path=<encoded-disk-path>`. When the user
clicks **Force Re-render All**, the server overwrites the atlas PNG on
disk — but the **URL is unchanged**, so the browser keeps using the cached
copy from the previous render until 24 hours elapse.

Timeline that produces the observed symptoms:

1. Earlier session generated atlas at `tile_px=96, cols=12`.
2. Browser cached that 1152 × 1152 PNG against the URL.
3. v9 deployed; `tile_px` defaulted to 192, `cols` to 8.
4. Force Re-render All → server writes new 1536 × 3264 atlas with
   tile_px=192. Cutout_index.json updated to match.
5. Dialog re-renders. UI reads cutout_index.json → sees `tile_px=192,
   cols=8`, computes CSS expecting a 1536 × 3264 atlas image.
6. Browser fetches the URL, gets a cache hit, returns the **old 1152 × 1152
   PNG** with the **old 96-px tile layout**.
7. My CSS positions and sizes the bg image as if it were 1536 × 3264 — but
   it's actually 1152 × 1152, scaled to fit the requested bg-size. Each
   "source tile" in the cached image is 96 px native, but it gets stretched
   into 1536/12 ≈ 128-pixel rendered tiles. None of the offsets line up.

Different display tile sizes (192, 96) and CSS approaches (pixel, %)
produce different *visible mismatches*, all caused by the same underlying
cache lie.

## 5. First thing the next session should do (5 minutes)

In a browser dev-tools network tab, open the candidate-preview dialog with
a known-good tomo, locate the request to `/api/vis-asset?path=…cutout_atlas.png`,
and check:

1. **Status:** is it `200 OK` or `200 (from disk cache)` / `304 Not Modified`?
   The latter two confirm the cache hypothesis.
2. **Response headers:** look for `Cache-Control: public, max-age=86400`.
3. **The response body's pixel dimensions** vs what the cutout_index.json
   says (`atlas_w × atlas_h`). Mismatch = cache lie confirmed.

Quick local override: try a hard refresh (Cmd-Shift-R / Ctrl-Shift-R) on
the page and reopen the dialog. If the gallery snaps to the right tiles,
the hypothesis is correct and the only remaining work is the cache fix.

## 6. The proper fix

Two complementary changes:

### 6a. Cache-bust the asset URL by mtime

In `ui/candidate_preview_dialog.py`, change `_vis_asset_url` so the URL
includes a query param keyed on the file's mtime:

```python
def _vis_asset_url(asset_path: str) -> str:
    try:
        v = int(Path(asset_path).stat().st_mtime)
    except OSError:
        v = 0
    return (
        "/api/vis-asset?path="
        + urllib.parse.quote(asset_path, safe="")
        + f"&v={v}"
    )
```

When the atlas regenerates, mtime changes, the URL changes, the browser
refetches. No stale cache. Same trick as `static/main.css` (see `main.py`
where `css_version` is computed from mtime).

The endpoint `serve_vis_asset` in `main.py:67` already ignores extra query
params (it only reads `path`), so no server change needed.

### 6b. Stop sending 24h cache for atlas/index files

`main.py:89` is too aggressive. Either:

- Reduce to `max-age=300` (5 minutes), so even unbusted URLs catch up
  during a session.
- OR send `Cache-Control: no-cache` for `*.json` (manifests change on
  every regen) and keep PNG cache so static assets (the WarpTools tomogram
  preview, which doesn't regen) can be cached.

Recommended: do **6a** for sure (it makes the system self-correcting on
regen); **6b** as a defensive belt-and-suspenders.

---

## 7. If cache-bust doesn't fix it

If hard-refresh + cache-bust still shows broken sprites, the problem is
elsewhere. Diagnostic order:

1. **Open the atlas PNG directly in a browser tab** (the URL from Network
   tab). Verify it actually shows an 8-col × 17-row grid of 192-px particle
   thumbnails. If the PNG itself looks wrong, the bug is in
   `render_pick_cutouts_atlas` (e.g., wrong tile placement).

2. **Inspect a tile div in DevTools.** Computed style: is `width` actually
   96 px? Is `background-size` actually `768px 1632px`? Is
   `background-position` what we set?
   - If computed is different from inline, something is overriding.
     Quasar/Tailwind class collisions are the usual suspect — search for
     other `.cb-gallery-tile` rules or any global `background-*` rules.
   - If computed matches inline but visual is wrong, suspect DPR / zoom
     interactions. Try at 100% zoom on a non-Retina display.

3. **Test with a single fixed tile.** Hardcode one `<div>` outside the
   gallery loop with the exact same style and see if it shows correctly.

4. **As a last resort:** drop the CSS-sprite approach entirely and use
   `<img>` tags clipped via a parent with `overflow: hidden` and negative
   `margin-top`/`margin-left`. More DOM nodes but avoids any
   background-image quirks.
