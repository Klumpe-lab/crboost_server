# Subtomo Gallery — How a sprite gets onto a tile

This is the canonical reference for **how each tile in the per-tomogram subtomo
gallery gets produced**, end-to-end. Read this if you're debugging "tile X is
missing", "the tile shows the wrong particle", "tiles are blurry/black", or you
just want to know what the gallery actually represents.

The gallery is generated by `services/visualization/preview_orchestrator.py`
+ `preview_render.py` + `subtomo_link.py`, and rendered by
`ui/candidate_preview_dialog.py`. Manifest version: **v9** (each version bump
auto-invalidates older caches).

---

## 1. The data flow at a glance

```
TEMPLATE_EXTRACT_PYTOM job          SUBTOMO_EXTRACTION job
  (the picks)                          (the 2D images)
        │                                      │
        ├── candidates.star ──► picks.json     │
        │   (Å coords, scores)                 │
        │                                      │
        │                                      ├── particles.star
        │                                      │   (Å coords, rlnImageName,
        │                                      │    rlnTomoVisibleFrames)
        │                                      │
        │                                      └── Subtomograms/<tomo>/<i>_stack2d.mrcs
        │                                          (per-particle 2D tilt stacks)
        │                                                      │
        │                                                      ▼
        │                          ┌──────────  subtomo_link  ─────────┐
        │                          │  builds {(tomo, Å-coord-key) →    │
        │                          │   {mrcs, visible_frames}}          │
        │                          └────────────────────┬───────────────┘
        ▼                                               ▼
  pick_coords (post-score-sort)  ──join via Å key──► pick_to_mrcs[i]
        │                                               │
        │                                               ▼
        │                            render_pick_cutouts_atlas()
        │                                  │
        │                                  ├── for each pick:
        │                                  │      _render_one_subtomo_thumb()
        │                                  │          mmap .mrcs → mean-of-visible-frames
        │                                  │          → percentile clip → resize to 192px
        │                                  │
        │                                  └── paste tiles into one PNG grid
        │                                      cutout_atlas.png  +  cutout_index.json
        ▼
  CSS sprite tile in the UI:
  background-image: url(cutout_atlas.png)
  background-position: -col*192px -row*192px
```

The gallery is fundamentally a **join** between picks (from the candidate-
extract job) and 2D images (from the subtomo-extract job). They share a
coordinate space but no foreign-key column, so we match on `(tomo_name,
rounded-Å-coord)`.

## 2. Step by step

### Step 1 — Read the picks

`generate_candidate_previews()` reads `candidates.star` from the
TEMPLATE_EXTRACT_PYTOM job. For each tomogram it pulls:

- `rlnCenteredCoordinateXAngst / Y / Z` — Å, origin = volume center, IMOD-up.
- `rlnLCCmax` (or fallback: `rlnAutopickFigureOfMerit`,
  `rlnMaxValueProbDistribution`) — the ranking score.

These are sorted score-descending. The post-sort index `i` is what every
downstream artifact (`picks.json`, the cutout atlas, the UI) uses to refer
to a pick.

### Step 2 — Build the subtomo lookup

`subtomo_link.build_pick_to_mrcs_index()` walks every job in
`project_state.jobs` whose type is `JobType.SUBTOMO_EXTRACTION`. For each
such job it reads `<job_dir>/particles.star`, which exposes:

- `rlnTomoName` — same identifier the candidate-extract job uses.
- `rlnCenteredCoordinateXAngst / Y / Z` — same coord system as the picks.
- `rlnImageName` — the path to the per-particle `.mrcs` (e.g.
  `External/job008/Subtomograms/<tomo>/42_stack2d.mrcs`).
- `rlnTomoVisibleFrames` — a string-encoded bit-vector like `"[0,1,1,0,...]"`,
  one entry per tilt frame, marking which frames the subtomo extractor
  considered usable.

We index by `(tomo_name, _coord_key(x, y, z))` where `_coord_key` rounds Å
to one decimal (RELION writes Å with one decimal in star files; this is
defensive, exact-match works in practice). When multiple subtomo jobs cover
the same pick (re-extractions, parameter sweeps), the lex-greater
relion_job_name wins — i.e., "newest extraction takes precedence" by
filename ordering.

### Step 3 — Join picks to .mrcs

For each pick (in post-sort order), we look up `(tomo, coord_key)` in the
index. Three outcomes:

- **Hit** → the pick maps to a `{mrcs, visible_frames, src_job_dir}`.
- **Miss** → no subtomo extraction processed this pick. Common causes: the
  candidate set was filtered between `template_extract` and `subtomo_extract`
  (e.g., score threshold), or the subtomo job is from a different
  candidate-extract instance.
- **Coord-key collision drift** → if RELION rewrote Å with different
  precision, the rounded key diverges. We've never seen this in practice but
  the failure surfaces as a miss.

`pick_to_mrcs[i]` is `None` on miss, `dict` on hit. The list is aligned
1:1 with `picks.json` order.

### Step 4 — Render one thumbnail

`_render_one_subtomo_thumb(mrcs_path, visible_frames, tile_px=192)`:

1. **Open the .mrcs** with `mrcfile.mmap()`. Each `.mrcs` is a stack of 2D
   images; the array shape is `(n_frames, H, W)` with H=W=448 in our test
   data (set by the subtomo extractor's box size).
2. **Apply the visible-frames mask.** If the `.mrcs` has 41 frames and
   `visible_frames` is `[0,1,1,0,...]`, we average only the frames marked
   `1`. If the mask is all zeros (RELION decided no frame is usable), we
   fall back to averaging all frames — better to show *something* than a
   black tile.
3. **Mean across visible frames** → 2D image `(H, W)` of float32. This is
   the operation that gives us a clean "what this particle looks like"
   image: stacking many tilts cancels noise.
4. **Mmap-view trap:** the slice has to be `np.array(..., copy=True)` while
   the `with mrcfile.mmap(...)` is still open, otherwise the mmap closes
   and the array becomes a dangling view. (See ROADMAP §3.5.)
5. **Robust contrast:** clip to the 1st–99th percentile of the mean image,
   then linearly scale to `[0, 255]`. Fixed min/max would wash out the
   particle on tomograms with rare extreme outliers.
6. **Resize** to `tile_px × tile_px` (default 192) with PIL Lanczos.
7. Return `(thumb_uint8_array, None)` on success, `(None, reason)` on
   failure.

### Step 5 — Pack the atlas

`render_pick_cutouts_atlas(pick_to_mrcs, atlas_path, index_path,
tile_px=192, cols=8)`:

1. Allocate a uint8 numpy array of shape `(rows * tile_px, cols * tile_px)`
   where `rows = ceil(n_picks / cols)`.
2. For each pick `i`:
   - If `pick_to_mrcs[i]` is `None`, record a failure with reason
     `"no subtomo match"` and skip.
   - Else call `_render_one_subtomo_thumb(...)`. On failure, record the
     reason (`"mrcs not on disk"`, `"mrc read error: ..."`,
     `"unexpected ndim 4"`, `"all-NaN frame mean"`).
   - On success, paste the thumbnail into the atlas at
     `(row, col) = divmod(i, cols)`, position `(row*tile_px, col*tile_px)`.
3. Save the atlas as a single PNG.
4. Save an index JSON listing `index[str(i)] = [row, col]` for every
   *successful* pick, plus the `failures` list, atlas dimensions, and
   `tile_px`/`cols` so the UI can compute background-positions without
   needing to redo the math.

If every pick fails, we still write the index JSON (with empty `index` and
the full `failures` list) so the UI can show the "why every tile failed"
expansion. The atlas PNG is omitted in that case.

### Step 6 — UI renders the gallery

In the dialog (`_render_gallery_body`):

1. Read `cutout_index.json`. The `"index"` dict maps post-sort pick index →
   `[row, col]` in the atlas.
2. For each pick (in the user-selected sort order — best/worst score, by Z):
   - Compute background-position: `-col * tile_px, -row * tile_px`.
   - Set background-size to the atlas's natural dimensions
     (`atlas_w × atlas_h`).
   - Render a div with that CSS, sized to `tile_px × tile_px` via
     `aspect-ratio: 1` and the grid's `repeat(auto-fill, minmax(192px, 1fr))`.
3. Failure list: `entry.cutout_failures` from the manifest is shown in an
   expansion below the grid, listing which picks failed and why.

The whole gallery scrolls inside its own container (`max-height: 75vh`) so
high-pick-count tomos don't push the dialog tall.

## 3. Why a tile might fail

Fastest read: open the "Why N tile(s) failed" expansion in the gallery card.
The reasons in order of how-often-we-see-them:

| Reason | What it means | Fix |
|---|---|---|
| `no subtomo match` | The (tomo, Å-coord) key from the pick wasn't found in any SUBTOMO_EXTRACTION job's particles.star. | Run subtomo_extract on the candidate set, OR check that the candidate set hasn't been filtered between extracts. |
| `mrcs not on disk` | The subtomo job's particles.star references a `.mrcs` path that doesn't exist. | The subtomo extractor crashed before writing; check the SLURM logs in that job's directory. |
| `mrc read error: <…>` | mrcfile threw on `.mmap()` — usually a corrupt header. | Re-extract that particular particle, or skip the file. |
| `unexpected ndim N` | The .mrcs is shape `(H, W)` (2D, single image) or 4D — handled but rare. We average single-frame .mrcs as-is; 4D fails. | Inspect the file with `header file.mrcs`. |
| `all-NaN frame mean` | Every visible frame was NaN at the particle position — happens for off-tomogram extractions. | Verify the pick coords are inside the tomogram dims. |

## 4. Caching behavior

Every output (atlas PNG, index JSON, picks.json) is mtime-checked against
its sources (`candidates.star`, `tomograms.star`). If sources are newer
than any output, the entry is regenerated. The manifest version (currently
**v9**) is the master cache key — older manifests are wholesale invalidated
on the next regeneration, so a v8 cache won't be reused after a v9 deploy.

To force a rebuild without touching sources: click **"Force Re-render All"**
in the dialog, or delete `vis/preview/manifest.json` from the candidate-
extract job dir.

## 6. Why this approach (vs alternatives)

**Why not render thumbnails on-demand at hover?** The gallery is meant for
*browsing* — sorting, scrolling, comparing. Lazy rendering means every
scroll fires N HTTP requests. Sprite atlas = one HTTP fetch, then pure CSS.

**Why not render thumbnails from the tomogram volume directly?** §3.1 of
ROADMAP.md: cryo-ET reconstructions have missing-wedge streak artifacts
that dominate any 2D projection at the per-pick scale. The whole point of
`relion_tomo_subtomo` is to extract clean per-tilt 2D images that the user
can actually evaluate. Re-projecting from the volume would give us blurry
streak-dominated stamps. The .mrcs frames are what the reconstruction
*literally measured at that location*, projected via the CTF and tilt
geometry RELION already worked out.

**Why match by coord instead of a foreign key?** RELION's job system
doesn't carry pick-IDs through job boundaries — each job rewrites its
inputs into its own particles.star with its own column ordering. The Å
coord is the only invariant, and it's exact-match (not approximate)
because RELION doesn't perturb coordinates downstream of the picker.
